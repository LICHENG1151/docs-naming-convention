# 问题记录文档命名规范

> 适用范围：`/Users/a1/work/work_space` 下记录**问题**的文档。
> 一篇问题记录 = 一个问题，记录其时间、背景、现象、分析与修复全过程。
> 命名格式**参照开发文档**，以日期开头，并与开发文档相互对应。

---

## 1. 文件命名

统一格式（与开发文档一致）：

```
<日期>-<当日序号>-<问题简述>.md
```

| 部分 | 规则 | 示例 |
|------|------|------|
| 日期 | 八位 `YYYYMMDD`，为问题**发现当天** | `20260701` |
| 当日序号 | 两位数字，从 `01` 起，同一天内递增 | `01` `02` |
| 问题简述 | 简洁中文短语，点出问题，不含空格 | `登录超时问题` `显存泄漏` |
| 扩展名 | 统一 `.md` | |

示例：

```
20260701-01-登录超时问题.md
20260701-02-显存泄漏.md
```

> 问题记录与开发文档共用 `日期-序号` 命名风格，靠**元信息头字段**互相关联（见第 5 节），不靠文件名前缀区分。

---

## 2. 元信息头

每篇文件**开头第一行起**，用 HTML 注释块记录管理信息：

```markdown
<!--
name: <问题简述，与文件名一致>
creator: Li Cheng
created: YYYY-MM-DD
modified: YYYY-MM-DD
related_dev: <关联的开发文档名，无则留空或写 -->
```

| 字段 | 含义 | 规则 |
|------|------|------|
| `name` | 问题简述 | 与文件名主题一致 |
| `creator` | 记录人 | 本目录默认 **Li Cheng**，他人记录改为实际姓名 |
| `created` | 发现时间 | 首次记录日期，创建后不再改动 |
| `modified` | 最后修改时间 | 每次更新（分析/修复）后改为当日 |
| `related_dev` | 关联开发 | 填对应开发文档名（可多个），见第 5 节 |

---

## 3. 问题状态（必填）

采用**最简两阶段**状态机，正文首个标题下用醒目一行标注：

```markdown
**状态**：未修复
```

| 状态 | 含义 |
|------|------|
| 未修复 | 问题已记录，尚未解决（可能仍在排查/修复中） |
| 已修复 | 问题已解决并验证 |

> 状态变更为 `已修复` 时，同步更新 `modified` 并填写修复时间与方案。

---

## 4. 正文结构（推荐，保持精简）

```markdown
# <日期>-<序号> · <问题简述>

**状态**：未修复

## 1. 背景与环境
时间、代码仓 / 分支 / commit、运行环境（硬件、依赖版本）等定位信息。

## 2. 问题现象
具体表现、报错信息、日志片段（贴关键部分即可）。

## 3. 复现步骤
如何稳定触发；若偶发，说明触发条件与概率。

## 4. 原因分析
排查过程与根因（排查中可持续补充）。

## 5. 解决方案
修复做法、涉及改动；填写**修复时间**。完成后状态改为「已修复」。

## 6. 关联开发文档
指向引入/修复该问题的开发文档（见第 5 节）。
```

原则：**时间、背景、现象**三项必写清楚，方便日后检索与回溯；分析与方案随进展补充。

---

## 5. 与开发文档的对应

问题记录与开发文档**双向关联**：

- 问题文档：元信息头 `related_dev` + 第 6 节指向开发文档名。
- 开发文档：元信息头 `related_issues` 列出本问题文档名。

示例对应：

```
问题文档  20260701-02-登录超时问题.md
              └─ related_dev: 20260701-01-登录接口重构.md
开发文档  20260701-01-登录接口重构.md
              └─ related_issues: 20260701-02-登录超时问题.md
```

> 若问题独立于任何开发任务（如线上偶发 bug），`related_dev` 留空；后续若为此专门开展开发，再补建开发文档并回填关联。

---

## 6. 新建问题记录速查

1. 命名：`<发现日期>-<当日序号>-<问题简述>.md`（序号当日从 `01` 递增）。
2. 顶部补元信息头，`creator` 默认 `Li Cheng`，`created`/`modified` 填当日。
3. 正文首行标 `**状态**：未修复`，先写清背景环境、现象、复现步骤。
4. 排查中持续补充原因分析。
5. 若关联某开发任务，双向填写 `related_dev` / `related_issues`。
6. 修复并验证后改 `已修复`，补充解决方案与修复时间。
