# 代码仓解析文档命名规范

> 适用范围：`/Users/a1/work/work_space` 下所有对代码仓进行结构解析、记录的文档。
> 目的：统一命名与元信息，保证任何人打开目录都能快速定位、了解文档归属与新旧。

---

## 1. 目录命名

每个被解析的代码仓，对应一个独立的文档目录，规则：

```
<代码仓名>-docs/
```

示例：`Megatron-LM` → `Megatron-LM-docs/`；`omniinfer` → `omniinfer-docs/`。

- 保持与源码仓库同名，仅追加 `-docs` 后缀。
- 一个仓一个目录，不混放。

---

## 2. 文件命名

统一格式：

```
<两位序号>[.子序号]-<中文主题>.md
```

规则说明：

| 部分 | 规则 | 示例 |
|------|------|------|
| 序号 | 两位数字，从 `00` 起，控制阅读顺序 | `00` `01` `02` … `11` |
| 子序号 | 需要在某主序号下细分时追加 `.数字` | `02.0` `02.1` `02.2` |
| 分隔符 | 序号与主题之间用连字符 `-` | `02-并行化子系统` |
| 主题 | 简洁中文短语，见名知意，不含空格 | `并行化子系统` `张量并行实现详解` |
| 扩展名 | 统一 `.md` | |

固定约定：

- `00-README.md`：每个 `-docs` 目录**必须**包含，作为导航总览，含阅读顺序表与一句话总览。
- `01-框架透视图解.md`：约定用于全局架构透视（各仓保持一致）。
- 主题需要拆分为多篇时，用父序号 + 子序号，例如 `02` 拆为 `02.0` / `02.1` / `02.2`，`02.0` 通常放前置基础。

示例（Megatron-LM-docs）：

```
00-README.md
01-框架透视图解.md
02-并行化子系统.md
02.0-Transformer与MoE结构基础.md
02.1-并行组构建与通信详解.md
...
11-整体逻辑综述.md
```

---

## 3. 管理信息（元信息头）

每篇 `.md` 文件**开头第一行起**，使用 HTML 注释块记录管理信息（不影响 Markdown 渲染）：

```markdown
<!--
name: <文档主题，与文件名主题一致>
creator: Li Cheng
created: YYYY-MM-DD
modified: YYYY-MM-DD
-->
```

字段规则：

| 字段 | 含义 | 规则 |
|------|------|------|
| `name` | 文档主题 | 与文件名中的主题一致 |
| `creator` | 创建人 | 本目录默认 **Li Cheng**，他人创建时改为实际姓名 |
| `created` | 创建时间 | 首次创建日期，格式 `YYYY-MM-DD`，创建后**不再改动** |
| `modified` | 最后修改时间 | 每次实质性修改后更新为当日日期 |

要点：

- 元信息头置于文件最顶部，紧接其后才是 `# 一级标题` 正文。
- `created` 一经写入保持不变；每次改稿只更新 `modified`。
- 多人协作时如需记录修改人，可在 `modified` 后追加 `by <姓名>`，例如 `modified: 2026-07-05 by Wang Fang`。

---

## 4. 正文结构建议（非强制，保持风格统一）

- 首个一级标题 `# <序号> · <主题>`，其后用引用块 `>` 写本篇定位、前置/后续文档链接、相关源码路径。
- 章节用 `##` / `###` 分层，交叉引用其他文档用相对路径链接：`[02.1](./02.1-并行组构建与通信详解.md)`。
- 涉及源码处标注文件路径（如 `transformer/transformer_layer.py`），便于回溯。

---

## 5. 新建文档速查

解析一个新代码仓时，按下列步骤：

1. 建目录：`<仓名>-docs/`。
2. 先写 `00-README.md`（导航）与 `01-框架透视图解.md`（架构总图）。
3. 按子系统依次 `02` `03` … 编号；单主题过大时用 `NN.M` 细分。
4. 每篇文件顶部补齐元信息头，创建人默认 `Li Cheng`，`created` / `modified` 填当日。
